Padrões de Projeto Aplicados (GoF)

A arquitetura de software do SyncMatch adota práticas avançadas de design patterns para desacoplamento de código e alta manutenibilidade.

Factory Method Creational

Implementado no cadastro de usuários via UserProfileFactory. Dependendo do tipo de conta selecionado no formulário, a factory gera o criador de perfil correto:

  • StudentProfileCreator: Cria registros na tabela candidate_profiles.
  • CompanyOwnerProfileCreator: Registra uma nova empresa e associa o usuário como admin proprietário.
  • RecruiterProfileCreator: Associa o recrutador a uma empresa existente como pendente de aprovação.
UserProfileFactory.php 
public static function make(string $role): ProfileCreatorInterface
{
    return match ($role) {
        'student' => new StudentProfileCreator(),
        'recruiter_new' => new CompanyOwnerProfileCreator(),
        'recruiter_existing' => new RecruiterProfileCreator(),
        default => throw new \InvalidArgumentException("Role inválida"),
    };
}

Strategy Behavioral

Utilizado para verificar a autorização de ações sobre vagas e candidaturas. O sistema injeta dinamicamente as classes de estratégia de acordo com as regras de negócio declaradas em ApplicationService:

  • AdminAuthorizationStrategy: Autoriza se o usuário for o administrador master do sistema.
  • CompanyOwnerAuthorizationStrategy: Autoriza se o usuário for o dono/proprietário da empresa criadora da vaga.
  • ApprovedRecruiterAuthorizationStrategy: Autoriza se o usuário for um recrutador ativo e aprovado pela empresa vinculada.

Observer Behavioral

Utilizado no fechamento automático de vagas. Quando uma candidatura é aprovada, o evento ApplicationApproved é disparado pelo service e o listener associado CloseJobOnApplicationApproval intercepta o sinal de forma assíncrona/desacoplada para fechar a vaga.

CloseJobOnApplicationApproval.php 
public function handle(ApplicationApproved $event): void
{
    $job = $event->application->job;
    if ($job && $job->status !== 'fechada') {
        $job->update(['status' => 'fechada']);
    }
}

Esquema do Banco de Dados

Abaixo estão descritos os principais campos e tipos de dados utilizados nas tabelas relacionais do sistema.

Tabela: users

Coluna Tipo Descrição
id BIGINT (PK) Chave primária autoincrementável.
name VARCHAR(255) Nome completo do usuário.
email VARCHAR(255) (Unique) E-mail de acesso.
password VARCHAR(255) Senha criptografada em Bcrypt.
role VARCHAR(50) Cargo do usuário (student, recruiter, admin, master).

Tabela: companies

Coluna Tipo Descrição
id BIGINT (PK) Chave primária da empresa.
name VARCHAR(255) Nome empresarial.
cnpj VARCHAR(25) Cadastro Nacional da Pessoa Jurídica.
user_id BIGINT (FK) ID do usuário dono / proprietário (admin) da empresa.

Mapeamento de Rotas e Middlewares

A proteção de rotas no SyncMatch é configurada no arquivo routes/web.php e implementada através de middlewares customizados.

Middlewares Customizados

  • CheckLogin: Intercepta as rotas autenticadas. Se o usuário não tiver uma sessão ativa (user_id), redireciona imediatamente para o /login.
  • CheckMaster: Middleware exclusivo. Garante que apenas usuários com cargo master possam acessar a prefixo de rotas /master/*.
  • CheckSession: Protege o formulário e a ação de login. Se o usuário já estiver logado, redireciona-o automaticamente para a página home.
